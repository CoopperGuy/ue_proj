# ARPG 구조 개선 1주 단위 실행 계획

이 문서는 현재 프로젝트의 고민 지점(GAS 운용, Room 구조의 데이터 주도화, GameMode/GameState 역할 정리)을 **작게 쪼개서 매주 실행 가능한 형태**로 정리한 계획이다.

---

## 목표 요약
- Room 구조를 "BP 상속 추가" 중심에서 "데이터 + 실행기" 중심으로 전환.
- GameMode 단일 의존을 줄이고, 전역 상태는 GameState로 이동.
- GAS는 유지하되, 게임 흐름 연동은 이벤트 기반으로 분리.

---

## Week 1 — 현재 구조 계측 + 의존성 정리 (리스크 없는 준비 주간)

### 목표
- 지금 구조를 안전하게 리팩토링하기 위한 "가시성" 확보.
- UI/Drop/Room 로직의 GameMode 직접 의존 지점 파악.

### 작업 항목
1. **의존성 맵 작성**
   - `GetAuthGameMode()` 호출 지점 리스트업.
   - `HUDWidget` 직접 접근 지점 리스트업.
   - Room 관련 BP/DA/Subsystem 연결 관계 다이어그램화.

2. **런타임 로깅 포인트 추가**
   - 룸 입장, 몬스터 스폰, 룸 클리어, 보상 지급 시점에 통일된 로그 태그 부여.
   - GAS 스킬 부여/실패 로그 포맷 통일.

3. **리팩토링 안전장치**
   - 핵심 플로우 체크리스트 작성:
     - 게임 시작
     - 룸 진입
     - 전투 시작/종료
     - 몬스터 드랍
     - 인벤토리 반영

### 완료 기준 (DoD)
- 위 5개 플로우를 재현하면서 로그로 순서 추적 가능.
- GameMode/HUD 직접 의존 지점 문서화 완료.

---

## Week 2 — GameState 도입 (상태 저장소 분리 시작)

### 목표
- "게임 진행 상태"를 GameMode가 아니라 GameState에서 관리.

### 작업 항목
1. **`ADungeonGameState` 생성**
   - 상태 예시:
     - `RunPhase`(Exploration/Battle/Reward)
     - `ClearedRoomCount`
     - `CurrentFloor`
     - `RunSeed`

2. **상태 변경 API 정의**
   - `SetRunPhase(...)`, `NotifyRoomCleared(...)` 등 단일 진입점 제공.
   - UI는 GameState 값/이벤트를 구독하도록 준비.

3. **읽기 경로 전환 1차**
   - "조회성" 정보(예: 현재 층, 클리어 룸 수)는 GameState에서 읽도록 변경.

### 완료 기준 (DoD)
- 플레이 중 상태값이 GameState에 일관되게 반영.
- 상태 조회를 위해 GameMode를 직접 참조하는 코드가 줄어듦.

---

## Week 3 — Room 데이터 주도화 1차 (규칙 데이터 분리)

### 목표
- 룸 행동 규칙(전투 시작 조건/문 제어/보상 트리거)을 데이터에서 주입 가능하게 전환.

### 작업 항목
1. **Room 규칙 데이터 구조 추가**
   - 예: `URoomRuleDataAsset`
     - SpawnRule (스폰 그룹 선택 방식)
     - DoorRule (잠금/해제 조건)
     - RewardRule (클리어 보상 정책)

2. **`ADiaRoomBase` 역할 축소**
   - 상태/실행기 역할 중심으로 유지.
   - 규칙 판단은 `RuleData`를 통해 수행.

3. **기존 BP_RoomA/B/C 이행 전략 적용**
   - BP별 하드코딩 차이를 우선 데이터로 승격.
   - BP는 비주얼/배치 차이 중심으로만 유지.

### 완료 기준 (DoD)
- 룸 타입 1~2개를 "데이터 변경만으로" 동작 차별화 가능.
- 새 룸 규칙 추가 시 C++ 수정 없이 대응 가능한 항목이 생김.

---

## Week 4 — Room 데이터 주도화 2차 (스폰/클리어 이벤트 정식화)

### 목표
- 룸 전투 생명주기를 명확한 이벤트 파이프라인으로 전환.

### 작업 항목
1. **룸 생명주기 이벤트 정의**
   - `OnRoomEntered`
   - `OnBattleStarted`
   - `OnWaveCleared`
   - `OnRoomCleared`

2. **몬스터 매니저/스폰 서브시스템 연동 정리**
   - "룸 GUID 기반" 보고 체계 표준화.
   - 클리어 판정 조건을 단일 경로로 통합.

3. **문/트리거 제어 일원화**
   - 문 잠금/해제 로직을 룸 이벤트 수신 기반으로 통일.

### 완료 기준 (DoD)
- 전투 시작/종료 흐름이 이벤트 로그로 명확히 보임.
- 룸별 예외 처리(특수 케이스) 코드가 감소.

---

## Week 5 — GameMode 책임 축소 + UI 결합도 완화

### 목표
- `GameMode -> HUD` 직결 경로 제거.
- UI 갱신은 Controller/UI Subsystem/GameState 기반으로 이동.

### 작업 항목
1. **UI 접근 경로 리팩토링**
   - 몬스터/인벤토리 컴포넌트가 GameMode에서 HUD를 직접 가져오지 않도록 변경.
   - PlayerController 또는 UI 전용 Subsystem 경유.

2. **드랍/알림 이벤트화**
   - 몬스터 사망 시 "드랍 요청 이벤트" 발행.
   - UI는 결과 이벤트만 구독.

3. **회귀 체크**
   - HP바, 인벤토리 반영, 전투 UI 표시가 기존과 동일하게 동작하는지 검증.

### 완료 기준 (DoD)
- GameMode는 규칙 오케스트레이션 중심.
- UI 관련 직접 의존(특히 월드 오브젝트 -> HUD) 현저히 감소.

---

## Week 6 — GAS 흐름 정리 (확장 가능한 구조로 고정)

### 목표
- GAS 자체는 유지하되, "스킬 효과"와 "게임 진행 상태"의 결합을 느슨하게.

### 작업 항목
1. **스킬 실행 결과 이벤트 분리**
   - 피해/상태이상/처치 결과를 Gameplay Event 또는 Delegate로 표준화.

2. **초기 스킬 부여 경로 안정화**
   - 초기화 타이밍(소유/복제) 점검.
   - 실패 시 디버깅 가능한 메시지 규격화.

3. **밸런스 데이터 테이블 운영 규칙 문서화**
   - SkillID, Tag, VariantID 네이밍/충돌 방지 규칙.

### 완료 기준 (DoD)
- 신규 스킬/변형 추가 시 코드 수정 범위가 최소화.
- GAS 디버깅 포인트(로그/태그)가 팀 내 공통 규약으로 정착.

---

## Week 7 — 안정화 + 기술부채 정리

### 목표
- 이전 주차 변경사항의 회귀 제거 및 문서/온보딩 마무리.

### 작업 항목
1. **회귀 테스트 시나리오 고정**
   - 기본 플레이 루프(입장→전투→클리어→보상) 점검표 확정.

2. **남은 임시 로직/주석 정리**
   - TODO, 임시 우회 코드, 중복 유틸 정리.

3. **운영 문서 업데이트**
   - Room 추가 가이드
   - GAS 스킬 추가 가이드
   - 상태 흐름(GameState) 가이드

### 완료 기준 (DoD)
- 신규 콘텐츠 추가 절차가 문서만으로 재현 가능.
- 핵심 루프에서 치명 회귀 없음.
